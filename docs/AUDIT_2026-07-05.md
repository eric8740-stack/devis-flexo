## 🔴 CRITIQUE

**C1 — La procédure de re-seed prod du README détruirait tous les tenants.**
`README.md:96-146` (rédigé à l'époque mono-tenant) demande de relancer
`python -m scripts.seed` contre la prod Railway. Or `scripts/seed.py:684-713`
fait des `DELETE` **globaux non scopés** (`session.query(Devis).delete()`,
`Entreprise`, `LotProduction`…). Suivre le README aujourd'hui effacerait les
données de **tous** les comptes.
→ Scoper tous les DELETE sur `DEMO_ENTREPRISE_ID` + garde-fou (refus sur
Postgres sans flag explicite) + réécrire la section du README.

## 🟠 ÉLEVÉE

**E1 — Bug produit : éditer un devis fait retomber Encres (P2) et Clichés (P3a) à 0 €.**
`crud/devis.py:1784-1786` : `update_devis` rechiffre les lots **sans passer
`nb_couleurs_par_type`** (contrairement au POST, fixé au Sprint 16,
`devis.py:397-403`). `DevisUpdate.nb_couleurs` est exposé mais jamais lu
(perdu dans un `setattr` sur attribut inexistant). Un devis 4+1 couleurs édité
voit son HT baisser silencieusement. Aucun test PUT n'asserte P2 > 0.
→ Mapper et propager `nb_couleurs` dans `update_devis` (fallback sur le
`payload_input` persisté) + test de non-régression.
*Lié : `audit_liaison_optimisation_devis.md` (27/05) signalait le même défaut
silencieux (`nb_couleurs_par_type={}`) sur le flux optimisation→devis + devis
à 0 € via complexe sans grammage — vérifier si toujours ouvert.*

**E2 — Contraintes UNIQUE globales non scopées tenant sur 8 tables : bloquant avant le 2ᵉ pilote.**
`machine.nom`, `tarif_encre.type_encre`, `tarif_poste.cle`,
`complexe.reference`, `operation_finition.nom`, `outil_decoupe.libelle`,
`partenaire_st`, `charge_machine_mensuelle(mois,annee)`. Conséquences réelles :
le 2ᵉ tenant qui choisit la même machine à l'onboarding prend un 409 ; un seul
tenant peut posséder les tarifs encre `pantone`/`process_cmj` → le 2ᵉ reçoit
`CostEngineError` sans pouvoir les créer.
→ Migration : composites `(entreprise_id, clé)`. Sans impact V1a.

**E3 — Fallbacks faibles si env Railway incomplète (à vérifier immédiatement).**
`auth_service.py:37-52` : sans `JWT_SECRET` → secret en dur
`"dev-secret-change-me"` (n'importe qui peut forger un token admin).
`scripts/seed.py:160-168` : sans `ADMIN_INITIAL_PASSWORD` → compte admin
`admin@devis-flexo.fr` / `admin`.
→ Vérifier les deux variables sur Railway ; faire échouer le démarrage/seed si
absentes.

**E4 — Bouton « Télécharger PDF » cassé : 401 systématique.**
`app/devis/[id]/page.tsx:148,201` + `DevisResultMultiLots.tsx:249` : lien
`<a href>` vers `GET /api/devis/{id}/pdf` qui exige un header Bearer qu'un lien
ne peut pas envoyer.
→ Fetch authentifié + blob (pattern déjà présent dans
`BatBlobAuthenticated.tsx`), à factoriser en `downloadAuthenticated()`.

**E5 — IDOR partiel : create/update devis ne valide pas l'appartenance tenant des FK.**
`crud/devis.py:288,1713` insèrent `client_id`/`machine_id`/`matiere_id`/
`cylindre_id` sans `validate_id_belongs_to_user` (contrairement à
`routers/cost.py:45-52`), et `_attach_relation_names` fait des `db.get()` non
scopés → un tenant peut lire les noms clients/machines/matières d'un autre en
devinant des IDs.
→ Valider chaque FK contre le tenant avant insertion.

**E6 — README massivement périmé + hygiène deps front.**
README à l'état Sprint 3 (annonce 14 tables/45 endpoints ; réalité 35/148) —
`docs/ETAT_PROJET.md` (12/06) est la bonne source. `shadcn` (CLI) en dependency
de **prod** → source des 10 vulns npm (`npm uninstall shadcn` + `npm audit fix`).
Next 14.2.35 : advisories high non corrigeables sans Next 15/16 (chantier dédié).

## 🟡 MOYENNE

- **M1** Rate limiting absent sur `/login`, `/register`, `/forgot-password` (brute-force, spam SendGrid).
- **M2** Refresh token 7 j non révocable, `/logout` no-op (pas de rotation/jti). Tokens en localStorage (XSS) — assumé MVP, à durcir post-pilote (cookie HttpOnly).
- **M3** `correspondance_laize_metrage` sans `entreprise_id` : un tenant modifie le mapping pour tous.
- **M4** `_calcul_laize_papier_lot` (`crud/devis.py:1350`) : `except Exception → None` → P1 retombe silencieusement sur le legacy (base de facturation matière changée sans log).
- **M5** Pages listes (devis, catalogue, clients, charges) : fetchs sans annulation → réponse périmée peut écraser la récente (pattern AbortController déjà présent ailleurs, à généraliser).
- **M6** God files : `lib/api.ts` 2 503 l (7 définitions de `API_URL`, refresh dupliqué), `devis/nouvelle/page.tsx` 1 452 l, `OptimisationRebobinage.tsx` 1 405 l.
- **M7** Dockerfile : root, pas de HEALTHCHECK. `requirements.txt` non lockés (`>=`) → builds non reproductibles ; pytest/pandas embarqués en prod.
- **M8** `.gitignore` ne couvre que `backend/*.db` → `devis_flexo.db` orphelin à la racine (repo public, risque `git add -A`).

## ✅ Ce qui est sain (vérifié)

- **Moteur de coût** : 100 % `Decimal` (aucun float sur l'argent), aucun fallback
  silencieux (raise explicites : config absente, grammage nul, vitesse nulle —
  division par zéro impossible), D1 bien conçu (dédup calage en aval, moteur intact).
- **Multi-tenant** : `scope_service.get_or_404_scoped` appliqué systématiquement
  sur tous les endpoints à id (vérifié router par router), 404 anti-énumération,
  `devis.numero` unique composite avec retry.
- **Auth** : JWT algorithme figé HS256, bcrypt 12 rounds, tokens email
  `secrets.token_urlsafe(32)` usage unique, anti-énumération login/forgot.
- **Uploads** : uuid4 + MIME whitelisté + anti path-traversal + limites de taille.
- **CI** : pytest complet sur Postgres 16 réel + garde-fou weasyprint ; front
  lint + vitest + build. `.dockerignore` propre, aucun secret commité.
- **Front** : 0 erreur TS, 0 `any`, ProtectedRoute au layout racine sans flash,
  gestion d'erreur systématique sur les listes, `next.config` ne masque rien.
